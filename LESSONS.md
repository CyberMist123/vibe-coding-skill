# Cross-Project Lessons

这里只保存经过真实项目或源码审计支持、可迁移到其他项目的经验。它不是某个项目的复盘全文，也不是通用口号集合。

每条经验必须包含：背景、规律、失败方式、建议做法、适用边界和证据级别。

## L001 — 一个项目只能有一个“当前事实”入口

**背景**  
同一个项目长期由多个 AI、窗口或 coding agent 接手时，README、Issue、handoff、TODO 和聊天记录容易分别保存不同版本的架构与进度。

**规律**  
当前需求、边界、架构索引、精确接口、运行流程、进度和下一步必须集中到一个权威文件。其他文档只保存各自负责的详细内容。

**失败方式**

- AI 从旧 Issue 读取过时命令；
- 文件存在就被误判为功能已完成；
- 新模型重新扫描整个仓库并创造第二套架构；
- 每次交接新增一个 `STATUS.md`，最终没人知道哪个有效。

**建议做法**

- 根目录使用 `PROJECT.md` 作为当前事实；
- `AGENTS.md` 只负责指向它并规定工作纪律；
- 功能改动完成后原地纠正旧事实，不追加第二份说明；
- Git 历史负责“过去发生过什么”，`PROJECT.md` 只负责“现在是什么”。

**适用边界**  
适用于持续开发、多人或多 AI 接手的项目；一次性小脚本可以简化。

**证据级别**  
已在 PI OS 项目中完成文档漂移收口，并支持新的执行窗口从权威入口接手后完成首次本地实例和手机/PC 网页链路；尚待第二个独立项目验证通用性。

## L002 — `AGENTS.md` 是入口，不是百科全书

**背景**  
为了防止 AI 乱改，项目容易把大量架构和历史全部塞进 Agent 指令，导致上下文膨胀、约束互相冲突并快速过时。

**规律**  
Agent 入口应短小，只说明：先读什么、允许做什么、绝对不能做什么、完成后如何同步事实。

**建议做法**

```text
AGENTS.md → PROJECT.md → 当前任务相关代码/文档
```

详细架构、部署与接口进入 `PROJECT.md` 的索引或专项文档。

**证据级别**  
已用于 PI OS 跨窗口接手；新执行窗口无需搬运完整设计会话即可继续首次部署。

## L003 — 必须区分“写了代码”和“真实验证”

**背景**  
AI 很容易看到脚本、测试文件或配置存在，就报告功能已经完成；但目标电脑、容器、网络或真实客户端可能从未运行。

**规律**  
项目状态至少分为：

- `计划中`
- `已实现/未验证`
- `已验证`
- `暂停`
- `放弃`

**建议做法**  
每个进度项同时记录证据：测试命令、日志、目标环境 smoke 或明确的人工确认。没有输出就不得升级为 `已验证`。

**证据级别**  
PI OS 曾将全部部署能力保持为“已实现/未验证”；目标电脑运行后，只把首次初始化和手机/PC 网页链路升级为“已验证”，发图、streaming、备份和自动启动仍保持待验收。

## L004 — 完工同步应基于事实差量，不基于会话摘要

**背景**  
普通 handoff 或 PR 总结往往记录“这次聊了什么”，但不能可靠说明项目当前架构、接口和剩余工作。

**规律**  
文档同步的证据只应来自：已确认需求、实际 diff、测试/验证输出和现有权威文档。

**失败方式**

- 把讨论中的候选方案写成已实现；
- 把长会话复制进项目文档；
- 只追加 CHANGELOG，不纠正当前架构；
- 接口重命名后旧命令仍留在 README。

**建议做法**  
先列出 `before → after` 的事实变化，再只更新受影响的文档，最后搜索本次变更造成的旧标识和矛盾。

**证据级别**  
已在 PI OS 的域名模型变更中使用：同步修改 PROJECT、架构、部署、恢复、Issue 和脚本接口。

## L005 — 重要不变量必须同时存在于代码防护和项目事实中

**背景**  
只在文档里写“不要改”不足以阻止自动化脚本或未来 AI 误操作；只在代码里拒绝也会让接手者不理解原因。

**规律**  
高风险不变量应具有双重保护：

1. `PROJECT.md` / `AGENTS.md` 明确说明边界和理由；
2. 实际脚本或代码在运行时校验并拒绝错误输入。

**PI OS 例子**  
`LOCAL_DOMAIN=pi.invalid` 同时写入项目事实、Agent 硬边界，并由 setup / 域名切换脚本检查。

**适用边界**  
身份标识、数据目录、密钥、生产环境、不可逆迁移和破坏性命令尤其需要。

**证据级别**  
代码防护已实现；目标实例已完成首次初始化，但该不变量的独立接口检查仍待记录。

## L006 — 项目结束时只回收“可迁移经验”，不要搬运项目私密历史

**背景**  
中央经验库若直接收录每个项目的完整复盘，会迅速变成难以搜索、泄露路径和业务细节的文档垃圾场。

**规律**  
Closeout 应把内容分成两类：

- 项目本地最终状态与复盘，留在项目仓库；
- 跨项目规律，去除名称、路径、密钥和一次性细节后提交到母包。

**可迁移经验格式**

```text
背景
规律
失败方式
建议做法
适用边界
证据级别
```

**证据级别**  
流程已设计，等待首个项目正式 closeout 验证。

## L007 — 跨窗口接手应传递“入口”，不要传递整段历史

**背景**  
设计、审计和执行常常发生在不同窗口。把整个旧会话复制给执行窗口会带来大量过期候选方案，也容易诱发重新审计。

**规律**  
旧窗口结束前应把已确认事实写入仓库；新窗口只从 `AGENTS.md` 进入 `PROJECT.md`，再读取当前任务直接相关的部署或接口文档。

**失败方式**

- 新窗口把已否决方案当成当前架构；
- 因上下文太长而跳过关键边界；
- 又做一次全库审计，延迟真实执行；
- 会话结束后进度没有落回仓库。

**建议做法**

```text
设计窗口：确认决策 → project-doc-sync → 结束
执行窗口：AGENTS.md → PROJECT.md → 当前任务文档 → 执行
执行完成：验证差量 → project-doc-sync
```

**适用边界**  
适用于长项目、多模型协作、设计与运行环境分离的任务。

**证据级别**  
已在一个真实自托管项目中使用；新的部署窗口从仓库入口接手后完成目标电脑启动及手机/PC 网页链路。

## L008 — 验证状态必须按能力切片，不能整项目连坐升级

**背景**  
一次成功通常只能证明一段链路。例如“手机能打开网页”能证明 DNS、Tunnel、反代和 Web 基础访问，但不能证明图片处理、实时流、备份或重启恢复。

**规律**  
进度表应按可独立验收的能力拆分。每条证据只升级它直接支持的项目，不使用含糊的“整体部署成功”覆盖所有子系统。

**失败方式**

- 网页可打开就把发图和 Sidekiq 标成通过；
- 登录成功就把备份恢复标成完成；
- 单元测试通过就声称生产环境已验证；
- 用户一句“好了”被扩写成全部 checklist 完成。

**建议做法**

- 将初始化、公网 HTTP、身份配置、登录、写入、媒体、streaming、备份和自动启动分开记录；
- 明确写出证据能证明什么、不能证明什么；
- 未覆盖的项目继续保持 `已实现/未验证` 或待验收。

**适用边界**  
适用于部署、迁移、支付、身份、安全、异步任务和多服务系统。

**证据级别**  
已在 PI OS 首次运行状态同步中实际使用：只升级首次初始化与手机/PC 网页链路，其余能力保持待验收。
