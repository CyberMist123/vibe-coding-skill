# 把连续性包交给 AI 时使用的提示词

## 新建项目

```text
请把随附的 project continuity kit 安装到当前项目，但先不要修改业务代码。

按以下顺序执行：

1. 阅读包内 kit/skills/project-bootstrap/SKILL.md。
2. 检查当前仓库已有的 README、AGENTS.md、架构/部署文档、Issue 和主要目录；只做建立当前事实所需的最小读取，不进行全库审计。
3. 将 kit/AGENTS.md、kit/PROJECT.md 和 kit/skills/ 复制或合并到项目根目录。
4. 若目标项目已有 AGENTS.md、PROJECT.md 或同名 Skill，不得直接覆盖；先比较职责，保留项目现有有效约束，再合并为一个入口和一个当前事实文件。
5. 根据用户已确认需求、实际代码和可验证输出填写 PROJECT.md：
   - 项目目标与验收条件
   - 已确认需求
   - 明确不做与安全边界
   - 当前架构、模块职责和数据流
   - 精确接口、端口、命令、环境变量、路径和数据所有权
   - 主要运行流程
   - 功能进度表
   - 当前下一步
   - 文档所有权图
6. 所有未知内容必须标为“未知/待确认”；不得猜测。
7. 状态只允许使用：计划中、已实现/未验证、已验证、暂停、放弃。
8. 没有真实测试、日志或人工 smoke 时，不得写“已验证”。
9. 完成后不要继续实现功能，只报告：
   - 创建/合并了哪些文件
   - 当前项目事实摘要
   - 仍然未知的事项
   - 建议的下一条可执行任务
```

## 已经开发一段时间、文档混乱的项目

```text
请使用随附的 project continuity kit 为当前项目建立唯一当前事实入口，不修改业务代码。

重点处理：

- 找出 README、Issue、handoff、TODO、架构文档之间互相冲突的当前说法；
- 以用户已确认需求、实际代码和验证输出为最高证据；
- 建立一个根目录 PROJECT.md；
- 让 AGENTS.md 只负责入口、边界和完工同步纪律；
- 保留详细专项文档，但在 PROJECT.md 中写清每份文档负责什么；
- 删除或标记过期的第二套状态/交接入口，不能让两个文件都声称自己是权威当前状态；
- 不重构代码，不修无关问题，不把历史会话复制进 PROJECT.md。

完成后执行 project-doc-sync 的漂移检查，并明确区分已实现/未验证和已验证。
```

## 日常任务

```text
先读根目录 AGENTS.md，按其中入口读取当前项目事实；只处理本任务，完成后执行项目文档同步。
```

## 项目或阶段结束

```text
先读 AGENTS.md 和 PROJECT.md，然后执行 skills/project-closeout/SKILL.md。

冻结最终进度和验证证据；把项目本地复盘留在项目仓库；另输出一段去除项目名称、路径、密钥和私密内容的“可迁移经验”，格式为：背景、规律、失败方式、建议做法、适用边界、证据级别。该经验将交回 project continuity kit 的 LESSONS.md 审核收录。
```
